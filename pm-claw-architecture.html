Layer 1 — Data Ingestion Input

This layer transforms the grey-box world (how PMs work today) into structured, queryable signals. Each connector normalizes raw data into a common event schema so the orchestrator doesn't care about the source.

Telemetry Connector

Streams product usage metrics, detects anomalies and trend shifts. Emits TELEMETRY_CHANGE events when thresholds are breached.

Source: internal dashboards, API catalog, NGC metrics

Feedback Connector

Aggregates internal user feedback + external user feedback. NLP classification into themes, sentiment, and urgency.

Source: surveys, Slack channels, support tickets

Issue Connector

Monitors NVBugs, forum posts, GitHub issues. Deduplicates, links to existing features, and tags severity.

Source: NVBugs API, GitHub API, forum scrapers

Market Intel Connector

Tracks competitor moves, enabling technology changes, and relevant market shifts. Periodic enrichment from market databases and news.

Source: market database, news feeds, analyst reports

Strategy Connector

Ingests OV core strategy changes and leadership directives. Translates high-level strategy into constraints the prioritization agent can use.

Source: strategy docs, leadership comms

Meeting Transcript Connector

Processes product pound and customer meeting transcripts. Extracts action items, feature requests, and commitments.

Source: meeting recordings, transcript services

Key Design Decision: Push vs. Pull

Telemetry and issues should be event-driven (push via webhooks) — you want to react fast. Market intel and strategy changes are lower-frequency and work better as scheduled polls (daily/weekly). Meeting transcripts are triggered on upload. The common event schema should include: source, timestamp, signal_type, urgency, raw_payload, extracted_entities.

Layer 2 — PM Orchestrator Brain

The orchestrator is the "PM brain" — it replaces the human PM as the central routing hub. It doesn't do the work itself; it classifies signals, maintains context, and dispatches tasks to specialist agents.

Signal Router

Classifies incoming events by type and urgency. Determines which specialist agent(s) should handle each signal. Can batch related signals together (e.g., 5 bug reports about the same feature → single prioritization task).

LLM classifier + rule-based urgency scoring

Context Store

Persistent memory that accumulates PM knowledge over time: feature history, customer relationships, past decisions and rationale, roadmap state. This is what makes the system smarter than a stateless pipeline.

Vector DB + structured knowledge graph (features ↔ customers ↔ priorities)

Task Planner

Given a classified signal + context, decides the action plan. May trigger multiple agents in sequence or parallel. Handles dependencies (e.g., "write user story" depends on "prioritize feature request").

LLM-based planner with DAG execution

Human-in-the-Loop Gateway

Controls which actions auto-execute vs. require PM review. Configurable per action type. Sends approval requests via Slack/email and waits for confirmation before proceeding.

Approval queue + notification integration

Layer 3 — Specialist Agents Workers

Each green box from your diagram maps to a specialist agent with a focused capability. They receive structured tasks from the orchestrator and produce concrete outputs.

📊 Reporter Agent

Generates telemetry change reports and integration status updates. Runs on schedule or triggered by anomalies. Outputs formatted reports (Slack, email, or dashboard).

🎯 Prioritization Agent

Takes feature requests, bug reports, and feedback signals. Scores them against strategy constraints, customer impact, and engineering effort. Outputs a ranked backlog.

📝 Doc Writer Agent

Writes and updates product documentation — API docs, integration guides, release notes. Can draft from scratch or update existing docs based on code changes.

🔧 Sample Builder Agent

Generates code samples and integration examples. Can create from API specs, adapt existing samples to new use cases, or build from customer requests.

🧪 Prototyper Agent

Builds quick prototypes and proof-of-concepts to validate feature ideas. Spins up minimal working demos that can be shared with stakeholders for feedback.

📋 User Story Agent

Converts prioritized features into well-structured user stories with acceptance criteria, ready for engineering sprint planning.

📢 Outreach Agent

Handles "actions to ping users" — sends targeted notifications to users about relevant feature updates, asks for feedback on specific topics, or follows up on reported issues.

🔄 Eng Tracker Agent

Monitors GitLab coding execution progress. Tracks commits, MR status, and pipeline results against the prioritized backlog. Flags slippage or blockers.

Autonomy Spectrum Governance

Not every action should auto-execute. Here's a suggested classification for each agent's outputs:

Agent Output Autonomy Rationale
Reporter Telemetry / status reports Auto Low risk — informational only, no side effects
Eng Tracker GitLab progress updates Auto Read-only monitoring, flagging only
Prioritization Ranked feature list PM Review Strategy-sensitive — PM should validate ranking before it's shared
User Story User stories for eng PM Review Directly shapes eng work — needs accuracy check
Doc Writer Documentation drafts PM Review External-facing — needs quality / accuracy review
Sample Builder Code samples PM Review Needs testing / correctness validation
Prototyper Working prototypes PM Review Stakeholder-facing — PM decides when to share
Outreach User notifications PM Approval Highest risk — directly contacts users, relationship-sensitive
Prioritization Strategy / roadmap guidance PM Approval Shapes product direction — cannot be fully delegated

Suggested Build Phases Rollout

Phase 1 — Read-Only Foundation (Weeks 1–4)

Build the data ingestion layer + Context Store. Get the Reporter Agent and Eng Tracker Agent working. These are low-risk, high-visibility wins — the PM claw starts producing value by surfacing information the PM would otherwise have to dig for manually. No external side effects.

Phase 2 — Prioritization Loop (Weeks 5–8)

Add the Prioritization Agent with PM-in-the-loop review. Connect to the backlog. This is where the system starts making judgment calls, so tight human oversight is critical early on. Calibrate against the PM's actual decisions.

Phase 3 — Content Generation (Weeks 9–14)

Add Doc Writer, Sample Builder, User Story, and Prototyper agents. All in PM-review mode. These amplify PM output — the system drafts, the PM edits and approves. Measure time savings.

Phase 4 — Outreach & Closed Loop (Weeks 15–20)

Add the Outreach Agent (PM-approval mode). Close the loop: the system can now ingest a signal, prioritize it, generate the artifacts, track execution, and notify users — with human checkpoints at key decision points.

Open Questions to Resolve Discussion

1. Single product or multi-product?

Does this PM claw manage one product, or should it be designed to handle multiple products from the start? This affects the Context Store schema significantly.

2. Where does the "Top 5" list come from?

Your data sources mention "Top 5" — is this a manually curated priority list? If so, does the Prioritization Agent replace it, or feed into it?

3. Eng interface: push or pull?

Does the PM claw push user stories directly into GitLab, or does eng pull from a shared backlog? Push is faster but requires more trust in the system's output quality.

4. How do you bootstrap the Context Store?

The system gets smarter with accumulated context. Do you seed it with historical data (past roadmaps, feature decisions, customer records), or start cold and let it learn?
